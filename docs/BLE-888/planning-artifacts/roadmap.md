## v0.2.0 — Growth（可靠性与开发者体验）

**目标**：让集成方在真实环境中信赖 SDK 的稳定性，并提升调试效率。

### 功能清单

#### 🔴 核心功能

| # | 功能 | 价值 | 复杂度 |
|---|------|------|--------|
| G1 | **离线指令队列** | 断线时缓存指令，重连后自动下发，避免"设置了但不生效" | 中 |
| G2 | **恢复出厂设置** | DPID 0x71，PRD Nice-to-Have，补齐协议完整性 | 低 |
| G3 | **低电量事件监听** | DPID 0x75，PRD Nice-to-Have，关键运维提醒 | 低 |
| G4 | **设备解绑** | 设备端操作上报处理，用户更换手机时需要 | 低 |

#### 🟡 开发者体验

| # | 功能 | 价值 | 复杂度 |
|---|------|------|--------|
| G5 | **BLE 调试面板** | Demo 中增加原始帧收发工具，手动输入十六进制帧调试 | 中 |
| G6 | **错误恢复引导** | `BlueError` 附带 `recoveryHint` 字段，告诉开发者下一步操作 | 低 |
| G7 | **连接状态可视化** | Demo 中实时显示状态机转换动画 | 低 |
| G8 | **连接日志导出** | 一键导出 SDK 日志文件，方便远程排查 | 低 |

#### 🟢 工程质量

| # | 功能 | 价值 | 复杂度 |
|---|------|------|--------|
| G9 | **BLE 模拟器集成测试** | CI 中跑 BLE 连接流程测试，不依赖真机 | 高 |
| G10 | **ProGuard 混淆验证** | 确认 AAR 混淆后 API 不丢失 | 低 |
| G11 | **版本号自动化** | fastlane / gradle-release-plugin 一键发版 | 中 |
| G12 | **XCFramework 打包脚本** | 生成可分发的 iOS 库文件 | 低 |

## v0.3.0 — Scale（多设备与响应式）

**目标**：支持护理院等多设备场景，提供现代化的响应式 API。

### 功能清单

| # | 功能 | 价值 | 复杂度 |
|---|------|------|--------|
| S1 | **多设备连接池** | 同时管理 N 个 LX-PD02 设备，每个独立状态机和指令队列 | 高 |
| S2 | **Kotlin Flow 扩展** | Android 响应式 API（独立模块，不增加核心 SDK 体积） | 中 |
| S3 | **Combine Publisher 扩展** | iOS 响应式 API（独立模块） | 中 |
| S4 | **设备分组管理** | 按房间/患者分组管理多设备 | 中 |
| S5 | **批量操作** | 一次性设置多设备闹钟（护士场景） | 中 |
| S6 | **连接优先级调度** | 多设备时按优先级分配 BLE 连接资源 | 高 |

## 优先级决策依据

**v0.2.0 为什么优先做"离线队列"而不是"多设备"：**
- 离线队列解决的是"单设备场景下的用户体验痛点"，影响 100% 用户
- 多设备是"特定场景的功能扩展"，影响 <10% 用户（护理院）
- 离线队列复杂度低（本地缓存+重连回调），多设备复杂度高（连接池+状态隔离）

**v0.3.0 为什么优先做"多设备"而不是"OTA"：**
- 多设备是 SDK 层可以独立完成的功能
- OTA 需要硬件方提供固件包格式、DFU 协议、升级流程——强依赖硬件方配合
- 多设备开放后可以进入护理院/医院市场，商业价值更高
